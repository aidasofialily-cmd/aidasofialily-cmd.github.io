OCamlBoy System Architecture

Hardware Abstraction Layer (HAL) & Module Interface Specifications

1. The HAL Architecture Strategy

In OCamlBoy, decoupled functional paradigms isolate execution cores from performance-bound platform environments (Native SDKs vs. WebAssembly target points). The interface abstracting CPU scheduling operations, video display bounds, audio generation pipes, and joypad mapping configurations relies on explicit signature layouts.

2. Core HAL Module Interface Specification (`hal.mli`)

This layout outlines the explicit abstract structure governing system backends:

(* OCamlBoy Hardware Abstraction Layer Signature Specifications *)

module type VIDEO_BACKEND = sig
  type t
  val render_frame : t -> Bigarray.int32_elt, Bigarray.c_layout) Bigarray.Array1.t -> unit
end

module type AUDIO_BACKEND = sig
  type t
  val queue_sample : t -> float -> float -> unit
  val clear_buffer : t -> unit
end

module type INPUT_BACKEND = sig
  type t
  type button = A | B | Select | Start | Right | Left | Up | Down
  val is_pressed : t -> button -> bool
end

(* System Core Functor combining execution abstractions safely *)
module type SYSTEM_HAL = sig
  module Video : VIDEO_BACKEND
  module Audio : AUDIO_BACKEND
  module Input : INPUT_BACKEND
end

3. Subsystem Sub-Allocation Maps

How individual abstraction interfaces resolve memory operations efficiently:

MMU & Memory Windows

Handles tracking layout states from High RAM (HRAM) registers to physical cartridge configurations using mutable tracking boundaries.

PPU Graphics Buffer

Extracts Game Boy LCD dimensions ($160 \times 144$) to a 32-bit linear pixel array utilizing blitting pipelines via native platform loops.

APU Sample Sync

Synchronizes high-fidelity waveforms across discrete audio sound channels dynamically with clock dividers to eliminate audio hitching.
